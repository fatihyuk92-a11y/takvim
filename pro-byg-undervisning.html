Setup & stack-valg

De fleste tutorials starter med "npm create next-app". Vi starter med beslutningen.

At bygge et produkt som praksis.ai betyder hundredvis af tekniske valg. De fleste har lille betydning. Få har enorm. Lad mig vise dig hvilke der er hvad — så du ikke spilder en uge på at vælge mellem Tailwind v3 og v4.

De fem valg der faktisk betyder noget.

  • Framework. Bestemmer hosting, performance, DX. Skift senere = uger.
  • Database. Bestemmer skalering, query-fleksibilitet, vendor lock-in. Skift senere = måneder.
  • Auth-løsning. Bestemmer compliance, brugeroplevelse, sikkerhed. Skift senere = brugerflugt.
  • Betalings-provider. Bestemmer fee-struktur, internationale priser, refunds. Skift senere = bogføringskaos.
  • Hosting. Bestemmer pris ved skala, deploy-flow, edge-performance. Skift senere = nedetid.

Alt andet — UI-bibliotek, form-validering, state management, analytics — kan skiftes på en eftermiddag uden konsekvens. Brug dine beslutningsenergi på de fem ovenfor.

Vores valg for praksis.ai — og hvorfor.

Next.js 14 (App Router)

Server Components, RSC streaming, indbygget i18n. Best i klasse for SEO-tunge content-sites. Alternativ: Astro hvis 95%+ statisk. Vercel-deployment under 30 sek.

Supabase

Postgres + auth + storage + edge functions i én pakke. RLS-policies giver multi-tenant gratis. Skifter til selvhostet Postgres på 1 dag hvis nødvendigt. Alternativ: Neon + Clerk hvis du vil splitte.

Stripe

Eneste seriøse valg for SaaS i 2026. Subscriptions, customer portal, tax-håndtering, EU-VAT moms. Lower fees end alternatives ved skala. Alternativ: Lemon Squeezy hvis du vil have Merchant-of-Record.

Vercel

Tightest Next.js integration. Edge functions globale på 1 klik. Preview-deployments per PR. Alternativ: Cloudflare Pages hvis du har stor static-traffic.

Tailwind v4 + shadcn/ui

Tailwind = ingen CSS-fil-håndtering. shadcn = copy-paste komponenter du ejer, ikke installerer. Skifte er ikke realistisk. Alternativ: Mantine eller bare CSS Modules.

Resend

Email der bare virker. React-templates, EU-server, billigt. Alternativ: Postmark hvis transactional, SendGrid hvis du elsker konfiguration.

Det vigtigste setup-valg: monorepo eller ej?

For praksis.ai: nej. Vi har én app. Monorepos (Turborepo, Nx) giver mening når du har 3+ apps der deler kode, eller stort team. For solo eller small team: én Next.js-mappe og en folder-struktur indeni.

Environment variables — den glemte fundament.

Den fælde alle falder i: hardcoded API-keys i koden. Eller en .env.local der har en typo og siloen for 3 dage. Lad mig vise dig den rigtige struktur fra start.

.env.local (lokal udvikling)
# Public — sendes til browser, må ikke være hemmelige NEXT_PUBLIC_SUPABASE_URL=https://xxx.supabase.co NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJ... NEXT_PUBLIC_SITE_URL=http://localhost:3000 # Server-only — ALDRIG NEXT_PUBLIC_ præfiks SUPABASE_SERVICE_ROLE_KEY=eyJ... STRIPE_SECRET_KEY=sk_test_... STRIPE_WEBHOOK_SECRET=whsec_... RESEND_API_KEY=re_... # AI-services OPENAI_API_KEY=sk-... ANTHROPIC_API_KEY=sk-ant-...
Sikkerheds-regel"Hvis det starter med NEXT_PUBLIC_, antag det er offentligt. Det vil ende i bundlen, i browseren, i Network-tab. Alt der skal være hemmeligt, må ikke have det præfiks. Hvis du er i tvivl, så er det IKKE NEXT_PUBLIC_."

Type-safe environment variables — den lille investering.

30 minutter en gang sparer 30 timer over et år. Validér environment variables ved opstart med zod:

lib/env.ts
import { z } from 'zod'; const envSchema = z.object({ NEXT_PUBLIC_SUPABASE_URL: z.string().url(), NEXT_PUBLIC_SUPABASE_ANON_KEY: z.string().min(1), SUPABASE_SERVICE_ROLE_KEY: z.string().min(1), STRIPE_SECRET_KEY: z.string().startsWith('sk_'), STRIPE_WEBHOOK_SECRET: z.string().startsWith('whsec_'), RESEND_API_KEY: z.string().startsWith('re_'), }); export const env = envSchema.parse(process.env); // Nu er process.env.SUPABASE_SERVICE_ROLE_KEY typet og garanteret findes

Hvor AI fejler

  • Vælger framework efter "hvad er hot". Stick med det der har et stort økosystem og lang levetid. Next, SvelteKit, Astro. Ikke det nyeste eksperiment.
  • Setup-paralyse. 3 dage på at vælge mellem Tailwind v3 og v4. Vælg, bygg, skift hvis nødvendigt.
  • Glemmer env-validation. App crasher i prod efter en uge fordi nogen tilføjede en variabel der mangler.
  • Hardcoder URL\'er. https://praksis-ai.vercel.app spredt rundt i koden. Brug NEXT_PUBLIC_SITE_URL.

Database & auth

Database-design er arkitektur. Auth er sikkerhed. Bland dem ikke sammen.

De fleste mistakes i SaaS sker i de første 100 linjer af database-skemaet. Du tilføjer en kolonne, så en anden, så bygger oven på et fundament der ikke kan bære. Lad os bygge praksis.ai's database fra bunden — én tabel ad gangen.

Skema-design — tænk i aggregater.

Hvert "ting" i din app er enten en hovedentitet (User, Course, Subscription) eller noget der hører til en anden. Det er forskellen mellem at have 7 tabeller og at have 47.

Praksis.ai skema (forkortet)
-- Brugere (Supabase auth.users er separat — vi bygger profil oven på) create table public.profiles ( id uuid primary key references auth.users(id) on delete cascade, email text unique not null, name text, created_at timestamptz default now(), -- Subscription-state synket fra Stripe stripe_customer_id text unique, subscription_status text check (subscription_status in ('trialing','active','past_due','canceled','none')) default 'none', subscription_tier text check (subscription_tier in ('free','intro','pro','none')) default 'none', current_period_end timestamptz ); -- Områder (statiske data der måske flytter fra MDX til DB senere) create table public.areas ( slug text primary key, name text not null, tier text not null check (tier in ('intro','pro')), num int not null, description text, display_order int ); -- Lektioner create table public.lessons ( id uuid primary key default gen_random_uuid(), area_slug text references areas(slug) on delete cascade, num int not null, title text not null, reading_minutes int, content_path text, -- sti til MDX-fil unique(area_slug, num) ); -- Bruger-fremgang (en række per bruger × lektion) create table public.lesson_progress ( user_id uuid references profiles(id) on delete cascade, lesson_id uuid references lessons(id) on delete cascade, status text check (status in ('not_started','in_progress','completed')), completed_at timestamptz, primary key (user_id, lesson_id) ); -- Index for hurtig "din fremgang"-query create index lesson_progress_user on lesson_progress(user_id, status); create index lessons_by_area on lessons(area_slug, num);

Row-Level Security — din vigtigste linje forsvar.

RLS er Postgres' indbyggede mekanisme der sikrer at brugere kun ser deres egne data — selv hvis din kode har en bug. Det er forskellen mellem en lille bug og en GDPR-katastrofe.

RLS-policies for lesson_progress
-- Slå RLS til alter table public.lesson_progress enable row level security; -- Brugere kan KUN se deres egen progress create policy "Users see own progress" on public.lesson_progress for select using (auth.uid() = user_id); -- Brugere kan opdatere deres egen create policy "Users update own progress" on public.lesson_progress for update using (auth.uid() = user_id); -- Brugere kan oprette nye records for sig selv create policy "Users insert own progress" on public.lesson_progress for insert with check (auth.uid() = user_id); -- INGEN delete-policy = ingen kan slette progress
RLS-mantra"En tabel uden RLS er en data-lækage der venter på at ske. Hver tabel, hver gang. Selv hvis du er sikker på at kun din egen kode skriver til den — RLS slået til, med policies."

Passwords er døde for nye SaaS. Magic links: brugeren indtaster email, modtager link, klikker, er logget ind. Mindre support, færre lækager, bedre UX.

Beskyttelse af ruter — middleware er din ven.

Du vil have /dashboard, /lektion/* osv. beskyttet. Den rene måde: Next middleware der tjekker session og redirecter ulogget brugere:

middleware.ts
import { createServerClient } from '@supabase/ssr'; import { NextResponse, type NextRequest } from 'next/server'; export async function middleware(request: NextRequest) { let response = NextResponse.next(); const supabase = createServerClient( process.env.NEXT_PUBLIC_SUPABASE_URL!, process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!, { cookies: { getAll: () => request.cookies.getAll(), setAll: (cookies) => cookies.forEach(({ name, value, options }) => response.cookies.set(name, value, options) ), }, } ); const { data: { user } } = await supabase.auth.getUser(); // Beskyttede ruter const protectedPaths = ['/dashboard', '/lektion', '/kursus']; const isProtected = protectedPaths.some(p => request.nextUrl.pathname.startsWith(p) ); if (isProtected && !user) { return NextResponse.redirect(new URL('/login', request.url)); } return response; } export const config = { matcher: ['/((?!api|_next/static|_next/image|favicon.ico).*)'], };

Tier-baseret adgangskontrol — hvor pro adskilles fra intro.

Intro-brugere skal ikke kunne se pro-lektioner. RLS klarer det igen:

Pro-content guard
-- Lektioner: alle kan se metadata (titel, beskrivelse) create policy "Anyone reads lesson metadata" on public.lessons for select using (true); -- Men content-funktionen tjekker tier create or replace function get_lesson_content(lesson_id uuid) returns text language plpgsql security definer as $$ declare lesson_tier text; user_tier text; content text; begin -- Hent lektionens tier (intro eller pro) select a.tier into lesson_tier from lessons l join areas a on a.slug = l.area_slug where l.id = lesson_id; -- Hent brugerens tier select subscription_tier into user_tier from profiles where id = auth.uid(); -- Pro-content kræver pro-abonnement if lesson_tier = 'pro' and user_tier != 'pro' then raise exception 'Pro-abonnement kræves'; end if; -- Returnér content-stien (eller faktisk content) select content_path into content from lessons where id = lesson_id; return content; end; $$;

Hvor AI fejler

  • Glemmer RLS på én tabel. Den tabel lækker. Tjekk hver tabel: alter table ... enable row level security.
  • Lagrer subscription-status kun i Stripe. Du betaler 100ms latency på hvert sideopslag. Sync til din DB via webhook.
  • Beskytter ruter kun client-side. Trivielt at bypass. Always server-side gennem middleware.
  • For mange tabeller. Hvis du har 30 tabeller dag 1, har du overengineered. Start med 5-7.
  • RLS slået til på hver tabel, med eksplicitte policies.
  • Magic links slår passwords i 2026.
  • Tier-check sker server-side i en SECURITY DEFINER-funktion.
  • Sync Stripe-status til din DB — ikke hent ved hver request.

Betaling & subscriptions

Stripe er den klogeste teknologi du kan integrere. Den er også den der bryder oftest.

Stripe er næsten perfekt — men der er tre steder hvor de fleste integrationer fejler: webhooks der mistes, subscription-status der går ud af sync, og refund-flows der mangler. Lad os bygge alt tre rigtigt.

De tre billing-modeller for praksis.ai.

Free

Begrænset adgang — måske 3 lektioner pr. område. Til at fange leads og lade brugere "smage".

Intro (166 kr/md)

Hele intro-biblioteket. 14 områder, 70 lektioner. Målgruppe: lærende der starter.

Pro (299 kr/md)

Intro + 6 pro-områder. Målgruppe: udviklere, founders, specialister. Årsabonnement 2.990 kr (spar 598).

Stripe-products opsætning.

I Stripe Dashboard: opret 4 prices (intro-monthly, intro-yearly, pro-monthly, pro-yearly). Gem price-id'erne i din kode:

lib/stripe/products.ts
export const STRIPE_PRICES = { intro_monthly: process.env.STRIPE_PRICE_INTRO_MONTHLY!, intro_yearly: process.env.STRIPE_PRICE_INTRO_YEARLY!, pro_monthly: process.env.STRIPE_PRICE_PRO_MONTHLY!, pro_yearly: process.env.STRIPE_PRICE_PRO_YEARLY!, } as const; export const TIER_OF_PRICE: Record = { [STRIPE_PRICES.intro_monthly]: 'intro', [STRIPE_PRICES.intro_yearly]: 'intro', [STRIPE_PRICES.pro_monthly]: 'pro', [STRIPE_PRICES.pro_yearly]: 'pro', };

Checkout-flow — Stripe Checkout slår custom forms.

Du kan bygge dit eget betalingsformular med Stripe Elements. Lad være. Stripe Checkout er optimeret af tusindvis af A/B-tests, håndterer 3D Secure, EU-VAT, alle kort-typer. Brug det.

app/api/stripe/checkout/route.ts
import { NextResponse } from 'next/server'; import Stripe from 'stripe'; import { createClient } from '@/lib/supabase/server'; const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!); export async function POST(req: Request) { const { priceId } = await req.json(); const supabase = await createClient(); const { data: { user } } = await supabase.auth.getUser(); if (!user) { return NextResponse.json({ error: 'Unauthorized' }, { status: 401 }); } // Find eller opret Stripe customer const { data: profile } = await supabase .from('profiles') .select('stripe_customer_id, email') .eq('id', user.id) .single(); let customerId = profile?.stripe_customer_id; if (!customerId) { const customer = await stripe.customers.create({ email: profile?.email, metadata: { supabase_user_id: user.id }, }); customerId = customer.id; await supabase .from('profiles') .update({ stripe_customer_id: customerId }) .eq('id', user.id); } // Opret checkout session const session = await stripe.checkout.sessions.create({ customer: customerId, mode: 'subscription', payment_method_types: ['card'], line_items: [{ price: priceId, quantity: 1 }], success_url: `${process.env.NEXT_PUBLIC_SITE_URL}/dashboard?welcome=true`, cancel_url: `${process.env.NEXT_PUBLIC_SITE_URL}/priser`, allow_promotion_codes: true, subscription_data: { trial_period_days: 7, // 7 dages gratis trial }, automatic_tax: { enabled: true }, // EU-VAT auto-håndtering }); return NextResponse.json({ url: session.url }); }

Webhooks — hjertet i din billing.

Stripe sender events når ting sker: kunde betaler, abonnement fornyes, betaling fejler. Du MÅ lytte til disse — ellers går din database ud af sync med Stripe inden for en uge.

app/api/stripe/webhook/route.ts
import { NextResponse } from 'next/server'; import Stripe from 'stripe'; import { createAdminClient } from '@/lib/supabase/admin'; import { TIER_OF_PRICE } from '@/lib/stripe/products'; const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!); export async function POST(req: Request) { const body = await req.text(); const signature = req.headers.get('stripe-signature')!; let event: Stripe.Event; try { event = stripe.webhooks.constructEvent( body, signature, process.env.STRIPE_WEBHOOK_SECRET! ); } catch (err) { return NextResponse.json({ error: 'Invalid signature' }, { status: 400 }); } const supabase = createAdminClient(); // service role, bypasser RLS switch (event.type) { case 'customer.subscription.created': case 'customer.subscription.updated': { const sub = event.data.object as Stripe.Subscription; const priceId = sub.items.data[0].price.id; const tier = TIER_OF_PRICE[priceId] || 'none'; await supabase .from('profiles') .update({ subscription_status: sub.status, subscription_tier: tier, current_period_end: new Date(sub.current_period_end * 1000), }) .eq('stripe_customer_id', sub.customer); break; } case 'customer.subscription.deleted': { const sub = event.data.object as Stripe.Subscription; await supabase .from('profiles') .update({ subscription_status: 'canceled', subscription_tier: 'none', }) .eq('stripe_customer_id', sub.customer); break; } case 'invoice.payment_failed': { // Send email til bruger, måske downgrade const invoice = event.data.object as Stripe.Invoice; // ... din logik break; } } return NextResponse.json({ received: true }); } export const config = { api: { bodyParser: false }, // VIGTIGT: Stripe har brug for raw body };
Webhook-fælde"Glem ikke at slå bodyParser fra. Stripe verifierer signaturen mod raw bytes — hvis Next parser JSON\'en først, vil signaturen aldrig matche. Du vil bruge 4 timer på at debugge dette uden hint."

Customer portal — lad Stripe håndtere det kedelige.

Brugere vil opdatere kort, ændre plan, downloade fakturaer. Bygg ikke det selv. Stripe Customer Portal gør alt — du sender bare brugeren derhen:

app/api/stripe/portal/route.ts
export async function POST() { const supabase = await createClient(); const { data: { user } } = await supabase.auth.getUser(); const { data: profile } = await supabase .from('profiles') .select('stripe_customer_id') .eq('id', user!.id) .single(); const session = await stripe.billingPortal.sessions.create({ customer: profile!.stripe_customer_id!, return_url: `${process.env.NEXT_PUBLIC_SITE_URL}/dashboard`, }); return NextResponse.json({ url: session.url }); }

Hvor AI fejler

  • Glemmer webhooks. Stripe og din database går ud af sync. Brugere mister adgang når de har betalt.
  • Stoler kun på client-side success. "Betaling lykkedes" i browseren betyder INTET. Vent på webhook.
  • Glemmer trial-håndtering. Brugere får adgang på trial, men du tjekker ikke current_period_end — de beholder adgang i 6 måneder gratis.
  • Tester ikke failed payments. Brug Stripe\'s test-cards (4000000000000341 fejler). Håndtér payment_failed event korrekt.
  • Stripe Checkout slår custom forms 99% af gangen.
  • Webhooks er ikke valgfri — ellers går DB ud af sync med Stripe.
  • Brug Customer Portal — byg ikke selv subscription management.
  • Test med Stripe CLI lokalt før du deployer.

Form-håndtering — react-hook-form + zod.

Komponenter — shadcn/ui er den moderne standard.

shadcn/ui er ikke et bibliotek du installerer. Det er komponenter du copy-paster ind i din kodebase — og dermed ejer. Du kan customize alt uden at fight et bibliotek.

Button, Input, Form

Forms-blokken. Med native accessibility, focus-states, error-states.

Dialog, Sheet, Popover

Modals der virker overalt. Accessible by default, Radix under hætten.

Toast (sonner)

Notifikationer der ikke er irriterende. Ind fra siden, forsvinder auto.

Card, Tabs, Accordion

Container-komponenter du bruger overalt.

Mobile-first — ikke valgfrit.

50%+ af din trafik bliver mobil. Hvis siden ikke fungerer perfekt på telefon, taber du halvdelen før de har læst en sætning. Test på din egen telefon dagligt — ikke kun dev-tools.

Performance — Core Web Vitals afgør SEO.

Google bruger LCP, FID, CLS som ranking-signaler. Mål og fix:

  • LCP under 2.5s — det største element skal være malet hurtigt. Brug next/image til al billede.
  • FID under 100ms — appen reagerer på input. Reducér JS i bundlen.
  • CLS under 0.1 — layout shifter ikke under load. Reservér plads til billeder/dynamic content.

Launch & iteration

De første 10 kunder afgør alt. Resten følger.

Du har bygget en kørende app. Tillykke. Nu kommer den sværeste del — at få brugere. Lad os tale om hvad der virker, og hvad der ikke gør.

Pre-launch tjekliste — de 12 ting før du åbner.

Pre-launch (gennemgå hver linje)
✓ Custom domain konfigureret + SSL ✓ Robots.txt + sitemap.xml ✓ Meta-tags (og:image, twitter-card) på hver side ✓ Privacy policy + Terms of service publiceret ✓ Cookie-banner (kun hvis du tracker) ✓ Email-domain konfigureret (SPF, DKIM, DMARC) ✓ Transactional emails testet (welcome, payment receipt, password reset) ✓ Stripe i live-mode med ægte test-betaling ✓ Webhook signing secret er PROD-versionen ✓ Error-tracking (Sentry) virker i prod ✓ Analytics (Vercel/Plausible) opsat ✓ Backup-strategi for DB (Supabase Daily Backup på)

Analytics — mål det rigtige.

De fleste tracker sidevisninger og kalder det "analytics". Det er bare støj. Mål de funnel-trin der betyder noget for forretningen:

Visit → Signup

Hvor mange besøgende opretter konto? Lav = forside er ikke overbevisende.

Signup → Trial activated

Hvor mange starter trial? Lav = trial-startup-friction er for høj.

Trial → Paid

Hvor mange konverterer? Lav = produktet leverer ikke nok værdi i trial.

Paid → Engaged

Hvor mange bruger appen ugentligt? Lav = onboarding/aktivering er svag.

Engaged → Retained (måned 3)

Hvor mange bliver? Lav = produktet er ikke "sticky" nok.

De første 10 betalende kunder — kommer fra dig.

Lad mig være helt ærlig: ingen ads, ingen SEO, ingen content-strategi vil bringe dig de første 10 kunder. Du får dem ved at tale med 100 mennesker en-til-en. Lad os gennemgå hvor.

  • Dit netværk. Skriv personligt til 30 personer du kender der matcher målgruppen. Ikke broadcast — en-til-en.
  • Communities. Reddit, Slack, Discord-grupper hvor målgruppen er. Bidrag i 2 uger før du nævner produktet.
  • Twitter/X i "build in public"-stil. Del processen, læringerne, fejlene. Skab et publikum mens du bygger.
  • Cold outreach. 20 personligt skrevne DMs pr. dag i 2 uger. 5% svar er normalt — det er 2 samtaler om dagen.

Iteration efter launch — tal med brugere.

De første 10 betalende kunder skal interviewes individuelt. Hver. Få dem på video-opkald i 20 minutter. Spørg:

Kundesamtale-skabelon
# Onboarding-samtale (20 min) 1. Hvad fik dig til at melde dig? - Specifikt udløsende moment? - Hvad prøvede du før praksis? 2. Da du loggede ind første gang: - Hvad var dit første indtryk? - Hvor langt klikkede du før du fandt værdien? 3. Hvilke 2-3 lektioner gav dig mest? 4. Hvis du skulle anbefale praksis til en ven: - Hvad ville du sige? - Hvilken type ven? 5. Én ting du ønsker var anderledes? # Lyt for: - De første ord de bruger (det er din positionering) - Hvor de tøvede (det er din friction) - Hvad de IKKE sagde (det er din blind spot)

Den første måned — fokus på retention.

De fleste startup-død kommer af dårlig retention, ikke dårlig acquisition. Hvis brugere kommer ind og forsvinder igen, kan du betale 1000 kr for hver ny kunde — det går ikke op.

  • Dag 1: Brugeren skal opleve et "wow"-øjeblik. Tilpas onboarding så det sker på 5 min.
  • Uge 1: Brugeren skal komme tilbage 3 gange. Tilpas notifikationer/emails.
  • Måned 1: Brugeren skal have skabt noget konkret. Spørg "hvad har du brugt det til?"
  • Måned 3: Brugeren skal være ambassadør. Spørg om de vil anbefale dig.

Iteration-rytme — weekly ship-cycle.

Ugentlig rutine i år 1
Mandag: Læs brugerfeedback fra forrige uge. Pick én ting at fixe. Tirsdag: Implementer fixét. Test grundigt. Onsdag: Ship til produktion. Annoncér ændringen. Torsdag: Tal med 2-3 brugere (calls eller chat). Fredag: Skriv en "build-in-public" post om ugen. Weekend: Hvile. Det øgede engagement holder ikke i år hvis du brænder ud. # Mål ugentligt: - New signups (mål: 10% vækst uge over uge) - Trial → Paid conversion (mål: 15%+) - Weekly active users (mål: 60%+ af betalende) - NPS af nye kunder (mål: 40+)
Sandheden om launches"Der findes ikke et "launch". Der findes hundredvis af mini-launches. Hver Hacker News-post, hver Twitter-tråd, hver pod-cast-interview er en \"launch\". Den første fejler nok. Den 47. får 1000 brugere. Bliv ved."

Hvor AI fejler

  • "Build it, they will come". Nej. Du skal aktivt finde dine første 100 brugere.
  • Stoler på ads tidligt. Du har ikke product-market fit endnu. Ads er bare dyrt feedback.
  • Optimerer signup-formularer i uge 1. Find først ud af om folk vil have produktet. Så optimer.
  • Læser ikke brugere. 3 måneder uden brugersamtaler = du bygger til en imaginær kunde.
  • De første 10 kunder kommer fra dig — en-til-en.
  • Mål funnel, ikke pageviews.
  • Tal med hver eneste af de første 50 betalende kunder.
  • Ship hver uge, mål hver uge.
